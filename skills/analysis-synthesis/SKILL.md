# Analysis Synthesis

검토·분석·조사 작업의 결과물은 보통 "발견 사항의 나열"에서 끝난다.
이 스킬은 그 나열 위에 **한 단계 높은 시점의 종합 평가**를 반드시 덧붙이게 한다.
개별 사실을 다시 요약하는 것이 아니라, 그 사실들을 **하나의 판단으로 묶어** 사용자가 "그래서 결론이 뭐고, 이걸 얼마나 믿어도 되는가"를 즉시 알 수 있게 하는 것이 목적이다.

## When to apply

분석 결과를 사용자에게 보고하는 작업의 **가장 마지막 단계**에서 적용한다.

- 적용: 코드 리뷰, 영향/사이드이펙트 분석, 원인 분석, 아키텍처 분석, 로그 조사, 보안 점검, 기술/버전 조사, 코드베이스 파악
- 미적용: 코드 작성·수정·실행 같은 산출 작업, "이 함수 시그니처가 뭐야" 같은 단일 사실 조회

판단 기준: 출력이 **여러 개의 발견 사항을 담고 있고**, 사용자가 그것을 근거로 의사결정을 해야 한다면 종합 평가를 붙인다.

## Output

본문(개별 발견 사항)을 평소대로 작성한 뒤, 맨 끝에 아래 형식의 블록을 추가한다.

```
## 종합 평가

**핵심 결론:** (1 ~ 2문장. 발견 사항 전체를 관통하는 단 하나의 판단.)

**심각도:** 상 / 중 / 하 — (왜 그 등급인지 한 줄 근거.)

**신뢰도:** 높음 / 보통 / 낮음 — (이 결론을 얼마나 확신하는지.)

**근거 한계:** (검증하지 못했거나 확인이 부족한 부분. 없으면 "없음"이라고 명시.)
```

길이는 분석 규모에 맞춰 조절하되 네 항목은 모두 채운다.

## Section guidance

### 핵심 결론 (필수)

- 개별 발견 사항의 재나열이 아니라, **그것들을 묶은 상위 판단**이어야 한다.
  - 나쁜 예: "버그 3건, 성능 이슈 2건을 찾았다."
  - 좋은 예: "기능은 정상이나, 동시성 처리가 단일 가정에 의존해 부하 시 깨질 구조다."
- 한 문장으로 말할 수 없다면 분석이 아직 결론에 도달하지 않은 것이다. 그 사실 자체를 결론으로 쓴다.

### 심각도 (필수)

- 발견 사항 중 **가장 무거운 것**을 기준으로 전체 등급을 매긴다.
- 등급은 상/중/하로 단순화하고, 반드시 한 줄 근거를 붙인다.
  - 데이터 손실·보안·운영 중단 가능 → 상
  - 기능 결함이나 유지보수 부담이지만 우회 가능 → 중
  - 스타일·사소한 개선 → 하

### 신뢰도 (필수)

- 결론을 내리기까지 본 근거의 **충분함**에 대한 자기 평가다.
- 코드/로그/문서를 직접 확인했으면 높음, 추론·일부 표본에 기댔으면 보통, 가정이 많으면 낮음.
- 신뢰도가 보통·낮음이면 "무엇을 더 보면 높아지는가"를 한 줄 덧붙인다.

### 근거 한계 (필수)

- 이번 분석에서 **다루지 못했거나 확인하지 못한** 영역을 솔직하게 적는다.
- 확인 안 된 것을 확인된 것처럼 쓰지 않는다. 한계가 없다고 판단하면 "없음"이라고 명시한다.

## Guardrails

- 종합 평가는 본문에 이미 나온 사실에 근거해야 한다. 평가를 위해 새 주장을 지어내지 않는다.
- 발견 사항이 0건이면 그것도 결론이다 — "검토 범위 내 문제 없음"을 핵심 결론으로 쓰고 신뢰도·근거 한계를 그대로 채운다.
- 긍정적으로 포장하지 않는다. 심각한 문제는 심각하다고 등급으로 말한다.
- 사용자 언어에 맞춘다. 한국어 요청이면 한국어로 작성한다.
